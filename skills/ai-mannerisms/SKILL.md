# AI Mannerisms to Avoid

1. **Unnecessary comments** -- don't restate what code already says. Comments explain *why*, not *what*.

2. **Unsolicited docstrings** -- skip for internal helpers and obvious methods. Docstrings belong on public APIs and genuinely complex functions.

3. **Filler phrases** -- cut "it's worth noting", "be sure to", "note that", "this ensures", "basically", "essentially", "in order to", "as mentioned above", "for the sake of clarity". Just say the thing.

4. **Aspirational adjectives** -- never call code robust, comprehensive, seamless, elegant, streamlined, scalable, modular, extensible, maintainable, clean, well-structured, battle-tested, production-ready, enterprise-grade.

5. **Hedging** -- "might potentially cause" / "could perhaps lead to". Be direct.

6. **Over-engineering** -- no error handling for impossible cases, config nobody asked for, abstractions for one-time operations, type annotations on obvious locals, backwards-compat shims for new code, feature flags that aren't needed. Three similar lines beat a premature abstraction.

7. **Em-dash overuse** -- rewrite the sentence instead. Usually a period and a new sentence is clearer.

8. **Uniform structure** -- don't force every bullet into the same grammatical template. Real writing varies.

9. **Commit/PR tells** -- no "This commit...", no file-by-file explanations, no `Co-authored-by:` trailers, no "Generated by" comments.

10. **Unnecessary local imports** -- use top-level imports unless there's a concrete reason (circular import, heavy module, `TYPE_CHECKING` guard).

11. **Writing from AI perspective** -- when writing text on behalf of the user (PRs, docs, reports), write as if they authored it. No "I analyzed...", "I implemented...".

12. **Passive voice overuse** -- "Redirect the user" not "The user is redirected". Prefer active voice and imperative mood.
