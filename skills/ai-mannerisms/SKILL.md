# AI Mannerisms to Avoid

## 1. Unnecessary comments on obvious code

Don't add comments that restate what the code says. Comments should explain *why*, not *what*.

```python
# Bad: comment restates the code
user = User.objects.get(pk=pk)  # Get the user by primary key

# Fine: no comment needed, the code is self-evident
user = User.objects.get(pk=pk)
```

Only add comments when the logic is non-obvious or there's a gotcha.

## 2. Unsolicited docstrings

Don't add docstrings to every function. Skip them for internal helpers, obvious methods, and anything where the name + signature says it all. Docstrings belong on public APIs and genuinely complex functions.

## 3. Filler phrases in prose

Strip these from docs, comments, commit messages, and PR descriptions:

- "It's worth noting that..." -- just say it
- "Be sure to..." / "Make sure to..." / "Note that..." -- just say it
- "This ensures that..." -- cut or rephrase
- "Basically..." / "Essentially..." / "In order to..." -- cut
- "As mentioned above/earlier..." -- cut
- "For the sake of clarity..." -- cut
- "It should be noted that..." -- cut
- "This is important because..." -- just say why

## 4. Aspirational adjectives

Never describe code or systems as: robust, comprehensive, seamless, elegant, streamlined, efficient, scalable, modular, extensible, maintainable, clean, well-structured, battle-tested, production-ready, enterprise-grade.

These are marketing words, not engineering words. If the code is good, the reader can tell.

## 5. Hedging language

Don't write "this might potentially cause issues" or "this could perhaps lead to problems". Either it does or it doesn't. Be direct.

## 6. Over-engineering

Don't add what wasn't asked for:
- Error handling for impossible cases
- Config options nobody requested
- Abstractions for one-time operations
- Type annotations on obvious locals
- Backwards-compatibility shims for code you just wrote
- Feature flags for features that don't need them

Three similar lines of code is better than a premature abstraction.

## 7. Em-dash overuse

AI-generated text is heavy on em-dashes. Don't just replace them with `--`; rewrite the sentence. Usually a period and a new sentence, or a comma, is clearer.

## 8. Suspiciously uniform structure

AI-generated text tends toward perfectly parallel bullet points, uniform sentence length, and consistent grammar patterns. Real writing varies. Don't force every list item into the same grammatical template.

## 9. Commit message and PR tells

Avoid in commit messages and PRs:
- "This commit..." / "This PR..." -- just describe the change
- Explanations of what every file does
- `Co-authored-by:` trailers for AI
- "Generated by" / "Created with" comments
- Overly long commit bodies that restate the diff

## 10. Unnecessary local imports

AI tends to put imports inside function bodies defensively, even when there's no circular import risk. Use top-level imports unless there's a concrete reason not to (circular import, heavy module you want to defer, `TYPE_CHECKING` guard).

## 11. Writing from the AI's perspective

When asked to write text (PR descriptions, docs, reports, emails), write from the user's perspective as if they wrote it themselves. Never use "I analyzed...", "I implemented...", "As an AI..." or similar. The output should read like the user authored it.

## 12. Passive voice overuse

"The user is redirected" vs "Redirect the user". Prefer active voice and imperative mood in docs and comments. Passive voice is fine occasionally but AI defaults to it excessively.

## 13. Dramatic phrasing

Avoid value-laden phrasing in docs, comments, commit messages, PRs, and chat output:

- "fails silently" / "silently swallows" / "dangerous" / "risky" / "unsafe"
- "journey" / "voyage" / "adventure"
- "speculated" / "we assumed" / "in theory"
- "comprehensive" / "exhaustive" / "battle-tested" / "production-ready"

State what the code does. If a function ignores errors, say "ignores `KeyError`", not "fails silently". If something has a tradeoff, name the tradeoff, not the vibe.

## 14. Silent deferral on multi-item asks

When the user says "do X for A, B, and C", do all three in the same response. Don't quietly skip C, declare the task done on A+B, and wait for the user to notice. If a step can't be done, say so explicitly. Never silently drop.

Applies to action plans, refactor sweeps, and "apply this pattern to every X" requests.

## 15. Response length

Match the answer to the question:

- Yes/no question mid-session → one sentence, no preamble
- Status check ("did it pass?") → one line, optionally followed by a short list
- Commit message → shortest valid message; expand only on request
- PR body for a trivial PR → one line
- Multi-step plan back from the user → just execute, don't restate

Long answers signal you didn't understand the question.

## 16. What to give back: text vs file

Match the artifact to the request:

- "Write me X" / "give me X" → write to a file (default to `tmp/<descriptive-name>` if no path given, and state the assumption)
- "What should this look like?" / "show me X" → inline in chat
- "Give me the test code" → executable code block, not prose explanation that requires a second prompt to extract

If the destination is ambiguous and matters, ask once. Don't dump a 400-line document inline when the user wanted a file.

## 17. Future/conditional tense in proposals and issues

When drafting issues, proposals, RFCs, or anything you haven't done yet, use "would" / "could" / "should", not "does" / "is" / "we built". Mixing tenses makes a draft look like a status update.

```
# Wrong (sounds like work is done)
The new cache layer is sharded by tenant. It uses Redis cluster.

# Right (proposal)
The new cache layer would be sharded by tenant. It would use Redis cluster.
```

Reserve present tense for things that actually exist.
